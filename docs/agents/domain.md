# Domain Docs

エンジニアリング系 skill がこのリポジトリを探索するとき、domain ドキュメントをどう参照するか。

## 探索前に読む

- **`CONTEXT.md`**（リポジトリ直下）— 用語集。規格の語彙の正名と避けるべき同義語。
- **`docs/adr/`** — これから触れる領域に関わる ADR。決定の背景・根拠・棄却案。
- **`spec/v1/spec.md`** — 規範仕様（RFC 2119 語彙）。規格として何が MUST / SHOULD / MAY か。

これらが無くても**黙って進める**。不在を指摘したり、先回りして作成を提案したりしない。producer skill（`/grill-with-docs`）が、用語や決定が実際に固まったときに遅延生成する。

## グロッサリの語彙を使う

出力が規格の概念を名指しするとき（issue タイトル・提案・仮説・テスト名など）、`CONTEXT.md` で定義された正名を使う。グロッサリが明示的に避けている同義語へ流れない。

必要な概念がまだグロッサリに無ければ、それはシグナルである。プロジェクトが使っていない語を発明しているか（再考する）、本当の欠落か（`/grill-with-docs` 向けに記録する）のどちらか。

## ADR との矛盾を明示する

出力が既存 ADR に反するときは、黙って上書きせず明示的に surface する。

> _ADR-0002（status 2 値）に反する。ただし〜という理由で再検討の余地がある。_
