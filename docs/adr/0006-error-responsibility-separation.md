# ADR-0006: item 起因エラーと全体エラーの置き場所を分ける

- ステータス: 採用
- 日付: 2026-07-05
- 関連: `spec/v1/spec.md`（§2, §3）, ADR-0002

> **2026-07-07 改訂注記（ADR-0011）**: 本 ADR の「item 起因エラーは item、item に紐づかない全体エラーは最上位 `errors[]`、両者を重複させない」原則は不変。複数主体化に伴い全体エラーが 2 層になった: 解決済み主体に紐づく全体エラー（例: その主体の lock 取得失敗）は `subjectResult.errors[]` に置き、最上位 `errors[]` は主体の列挙・解決の前段のエラー（入力 parse 失敗・主体列挙自体の失敗等）のみに限定される（→ ADR-0011）。

## 背景

エラーには item に紐づくもの（ある処理単位が失敗した）と、item に紐づかない全体エラー（入力 parse 失敗・lock 取得失敗）がある。両者の置き場所を決めないと、同じエラーが複数箇所に重複して型と意味が閉じない。

## 決定

- item に紐づくエラーの一次情報源は item 自身（`result.items[].error`）とする。
- 最上位 `errors[]` には **item に紐づかない全体エラーのみ**を置く（入力 parse 失敗・lock 取得失敗等）。item 起因のエラーをここに置いてはならない。
- 同じエラーを両方に重複させない。

## 根拠

- 全 item エラーの集約ビューが欲しければ `jq` で `result.items` から導出できる。エンベロープに冗長な集約ビューを持つのは過剰。
- `result.items` の責務を「宣言された処理単位の結果」に限定することで、型と意味の両方が閉じる。

## 影響

- `spec/v1/spec.md` §2（最上位 `errors[]`）・§3（item の `error`）。
- `schema/v1/envelope.schema.json` の `errors[]` と `items[].error`。
- `testdata/v1/invalid/` に「item 起因エラーを最上位 `errors[]` に置く」例を拒否するベクタを置く。

## 棄却した代替案

- **全 item エラーを最上位 `errors[]` にも集約する**: 冗長で二重管理になり、item と errors の整合を保つ責任が増える。
- **`errors[]` を廃止して全て items に置く**: item に紐づかない全体エラー（入力 parse 失敗等）の置き場所が無くなる。
