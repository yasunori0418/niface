# ADR-0020: skipped を前段の失敗による未実行に限定する

- ステータス: 採用
- 日付: 2026-07-07
- 関連: `spec/v1/spec.md`（§3）, ADR-0002, ADR-0005, ADR-0007, ADR-0009

## 背景

item の `status: "skipped"` は spec §3 で「『前段の失敗による未実行』**等**に使う」と定義されており、「等」により適用範囲が開いていた。nherd の restart 不要判定や nshadow の「ユーザー削除をしない」方針のように、失敗起因でない意図的な不作為を `success` + `info` と `skipped` のどちらで表すかがツール毎に割れる余地があり、表現が不統一になると消費側の解釈が壊れる。また skipped の理由を運ぶ標準スロットも無い（`error` は failed 時のみ必須）。（#13）

## 決定

- `skipped` は「**前段の失敗による未実行**」にのみ使う。適用範囲を単一の因果（依存する前段の失敗）に限定し、「等」を廃する。
- 方針による不作為（restart 不要の判定、削除をしないという方針等）は**判定処理の成功**であり `status: "success"` で表す。背景・理由が必要なら `W_` 警告（→ ADR-0005）または `info`（→ ADR-0007）で運ぶ。
- schema は無改訂（語彙の明確化のみ）。specVersion は据え置く。
- 「dry-run では使わない」（→ ADR-0009）は不変。skipped が前提とする「前段の失敗による未実行」は実際の実行過程でのみ生じ、dry-run の items（「実行したら何をするか」の宣言）とは両立しない。両者は整合する。
- skipped の理由を運ぶ標準 reason スロットは今回追加しない。需要が実証されたときの互換追加として温存する。

## 根拠

- 意味を単一の因果に固定すると、消費側は「skipped がある ⇒ その主体の実行のどこかに失敗がある」と機械的に推論できる。方針スキップが混ざるとこの推論が成立せず、skipped の解釈に per-tool 知識が要るようになる。
- 方針による不作為は「何もしない」と**判定する処理が正常に完走した**結果であり、item status の語彙上は success が正しい。不作為の理由はツール固有の文脈なので `info` の領分（→ ADR-0007）。
- 語彙の明確化だけで不統一の余地を消せるため、schema 変更（新 status 値・新フィールド）という高コストな手段を要しない。

## 影響

- `spec/v1/spec.md` §3 の skipped 記述の限定明確化のみ。schema / go / testdata は無改訂。
- `docs/guides/change-kind-mapping.md`（#14）の「restart 不要判定 = `success` + `info`」指針が本決定を前提にする。
- `CONTEXT.md` の item 項に skipped の語彙を同期する。

## 棄却した代替案

- **status の enum に新値を追加する（`unchanged` 等）**: 未知フィールドの must-ignore（→ ADR-0010）は**未知の enum 値**を救えない。既存消費側の status 分岐（3 値前提の網羅）を壊すため実質非互換であり、互換追加を装えない。語彙の明確化で足りる。
- **標準 reason スロット（skipped 理由の共通フィールド）を追加する**: 「2 つ以上のツールで同じ意味の code が必要」という needs 駆動の追加条件（→ ADR-0005 の精神）に相当する需要が未実証。理由は当面 `info` で運べ、需要が実証されたら互換追加できるため、今決める必要がない。
