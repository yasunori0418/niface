# ADR-0014: 規格の領分を結果エンベロープに限定し世代表現を独立スキーマにしない

- ステータス: 採用
- 日付: 2026-07-07
- 関連: `docs/concept.md`, `docs/ecosystem/overview.md`, `docs/ecosystem/distro-plan.md`, ADR-0004, ADR-0005, ADR-0011

## 背景

concept.md は niface を「stdout / stdin を通じて会話するための共通 JSON 規格」と述べ、distro-plan.md は横断規格を「世代表現・差分表現・結果エンベロープ」の 3 点と定義してきた。しかし spec v1 が実際に定義しているのは結果エンベロープ（差分表現 `changes[]` を含む）のみで、入力 JSON の形・世代表現・rollback 呼び出し規約・複数ツール集約の形状は存在しない。この不一致を放置すると、将来の作業者が「入力規格や世代表現は未実装の宿題」と誤読して規格を肥大させる方向に働き、M2（nboot）以降の実装が曖昧な契約の上で始まる。planned 5 ツールの出力シミュレーションによる再検査（2026-07-07・tracking #8）を受け、niface の領分を確定する。

前提として、各ツールは nix-env 経由のプロファイル化を予定しており（nput は実装済み）、profile 世代 = ビルド済み宣言（closure）のスナップショット列が全ツールで手に入る。

## 決定

- niface の領分は**結果エンベロープ**——各ツールの実行結果を漏れなく収集し構造化情報として扱うためのデータ構造——のみとする。
- 以下は niface の領分外とし、規格に持ち込まない:
  - **入力 JSON の形**: 各ツールが自リポジトリで公開・管理する（`info` スキーマと同じ構図）。spec §8 の standalone 入力規律（stdin の JSON か明示引数のみ）は入力の**受け取り方**の規律として維持する
  - **rollback の呼び出し規約**: ncompose と各ツールの領分。niface が担うのは判断材料（`changes` / `reversible` / 適用済み差分）の完全性のみ
  - **複数ツール集約の形状**: ncompose の領分（→ ADR-0011）。現行規格の item（1 ツール実行）+ `info`（子エンベロープ）で表現できる
- **世代表現は独立スキーマとして持たない**。世代は（profile 世代番号, その世代を適用 / 計画したエンベロープ）の対で表し、世代間の中身の差分は id 安定性（→ ADR-0004）によるエンベロープ間の items 突合で導出する。
- apply の意味論を「現実を宣言世代 N に一致させる調停」と捉える。rollback は「旧世代の宣言の再適用」であり、`reversible` は「旧宣言の再適用で戻れるか」を表す。
- 世代遷移の報告語彙（標準スロット）と実行相関キーは、本決定と独立の互換追加として別途扱う（#17 / #18）。

## 根拠

- 契約の置き場が「結果 = niface 規格、入力の形 = 各ツール公開スキーマ」と確定することで、ncompose・nboot が何に依存してよいかが明確になり、規格がツールのリリースサイクルに引きずられない（concept.md「ツール固有スキーマを管理しない」の徹底）。
- id 安定性（ADR-0004）が既に「世代 N と N+1 の差はエンベロープ同士の id 突合で取れる」ことを保証しており、世代表現スキーマを別に発明する必要がない。エンベロープ自体が世代のマニフェストとして機能する。
- 「宣言世代への調停」という意味論は、profile 切替が現実の適用を意味しない道具（nboot の ESP コピー・nwrap の tmpfs 再生成・nshadow の passwd 調停）まで一般化でき、ツール毎の undo 実装を不要にする。ブートメニューからの旧世代起動も稼働中 rollback も「世代 N への調停」という同一操作に還元される。
- システム世代（ブートメニューの 1 行）= 各ツール世代の整合した組、という合成は ncompose の領分であり、niface は相関キーの置き場（#18）だけ提供すれば収集エンベロープ列から再構成できる。

## 影響

- `docs/concept.md`: 冒頭の「stdout / stdin を通じて会話するための共通 JSON 規格」を結果エンベロープ規格である旨に改め、「これがしないこと」へ入力の形・世代表現スキーマ・rollback 呼び出し規約を追記。
- `docs/ecosystem/overview.md`: 設計原則 3「規格が契約」を「結果の契約は niface、入力の形は各ツール公開スキーマ」に精緻化。
- `docs/ecosystem/distro-plan.md`: 横断規格 1 の「世代表現・差分表現・結果エンベロープ」と rollback プロトコル節を本決定に同期。
- `CONTEXT.md` / `CLAUDE.md` / `README.md` / `README.ja.md`: 冒頭の説明文を同期。
- `spec/v1/spec.md` / `schema/v1/`: 変更なし（既に結果エンベロープのみを定義しており本決定と一致する）。

## 棄却した代替案

- **入力規格（最低限、世代表現）を niface に追加する**: 「規格が契約」を入力側にも広げる案。しかし世代一覧は producer（nput）の出力であり、その形は nput が自リポジトリで管理する構図が既に成立している。niface に持ち込むと規格がツールのリリースサイクルに引きずられ、concept.md「ツール固有スキーマ（info の中身）を管理しない」と矛盾する。
- **世代表現の独立スキーマ（generations.json の規格化）**: エンベロープと別に世代一覧の型を規格が持つ案。id 安定性により世代の中身はエンベロープで表現でき、二重のマニフェスト形式は同期義務だけを増やす。
- **rollback 呼び出し規約の規格化**: 呼び出し（コマンド名・引数・入力の与え方）は実行時プロトコルであり、データ構造の規格に混ぜると領分が崩れる。ncompose 設計時に ncompose 側の文書で定め、niface は判断材料の完全性（#11 等）で支える。
