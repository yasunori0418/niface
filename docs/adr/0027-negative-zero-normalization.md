# ADR-0027: identity の負ゼロ（`-0`）を `0` に正規化して域内とする

- ステータス: 採用
- 日付: 2026-07-18
- 関連: `spec/v1/spec.md`(§5・§2), `testdata/v1/id-vectors.json`, `go/id.go`, `nix/lib.nix`, ADR-0004, ADR-0010, ADR-0024
- 改訂対象: ADR-0010（specVersion 据え置きの互換扱いに「全実装が既に一致した挙動の規範化」類型を追加。ADR-0024 が第 1 類型を足したのに続く精緻化）

## 背景

ADR-0024 は identity の数値を「整数表記の整数（±(2^53−1)）」に限り、数値は値でなく表記で判定すると定めた。しかし `-0`（負ゼロ表記）が域内か域外か、域内なら `0` と同一 identity か別 identity かを spec §5 は明示していなかった。`-0` は valid JSON の整数表記であり、consumer が key に書き得る。

両参照実装は既に `-0` を `0` として扱っている（実測済み）。

- Go(`go/id.go`): `encoding/json` の `UseNumber` デコードで `-0` は `json.Number("-0")` になる。`canonicalize` は小数点・指数（`.eE`）を含まないため整数表記と判定し、`strconv.ParseInt("-0", 10, 64)` が `0` を返し、`strconv.FormatInt(0, 10)` で `"0"` を出力する。
- Nix(`nix/lib.nix`): `builtins.fromJSON "-0"` は整数 `0` を返し、`deriveId` の `builtins.toJSON` が `"0"` を出力する。

つまり両実装とも canonical が `0` に収束し id は一致するが、その一致は spec の規範ではなく実装挙動の暗黙の合意に留まっていた。id-vectors に `-0` の受理ベクタを追加する（issue #40 ④）にあたり、この暗黙合意を規範化する必要が生じた。

## 決定

- `-0`（負ゼロ表記）を identity の**域内**とし、`0` と**同一の identity** として扱う。実装は id 導出時に `-0` を `0` へ正規化しなければならない（MUST）。これを spec §5 の数値項に明文化する。
- 参照実装のコード（`go/id.go` / `nix/lib.nix`）は変更しない。両実装は既にこの正規化を行っており、本 ADR は spec を既存挙動に追いつかせる明文化である。
- `testdata/v1/id-vectors.json` に `-0` の受理ベクタ（`kind: "negzero"`, canonical `0`）を追加し、Go / Nix 双方がこの正規化で id を一致させることを言語横断で固定する。
- 本変更は必須化（`-0` 正規化 MUST の追加）を伴うが specVersion を上げない。全実装が既に一致した挙動を規範化するだけで id 導出結果も消費側の観測値も変わらないためである。この据え置き根拠を spec §2 の互換ポリシーに第 2 の据え置き類型として明文化する（ADR-0024 が足した第 1 類型に続く精緻化。→ ADR-0010）。

## 根拠

- `-0` を**域外拒否**にせず**域内正規化**にするのは、整数表記に符号付きゼロの区別が存在しないためである。IEEE 754 浮動小数の `-0.0` とは異なり、整数の `-0` は `0` と数学的に同一で、両実装のパーサが自然に `0` へ畳む。ここで拒否を課すと、実装に「正規化してから id を出す」現状の素直な経路をわざわざ「`-0` を検出して弾く」特殊分岐に置き換えることになり、実装が複雑になるだけで互換上の利得がない。
- 域内正規化は ADR-0024 の「表記で判定する」モデルと矛盾しない。`1.0` を域外にするのは、`1`（整数正準形 `1`）と `1.0`（float 正準形が実装依存で割れ得る）が別 id に落ちる危険があるためだった。`-0` にはこの危険がなく、両実装の整数パーサが決定的に `0` へ収束する。
- 小数表記の負ゼロ `-0.0` は別扱いである。`-0.0` は小数点表記のため ADR-0024 の規則で既に域外（Nix は `builtins.fromJSON "-0.0"` を float にし deriveId が拒否、Go は `.` を含むため `errNonInteger`）。本 ADR が域内とするのは整数表記の `-0` のみ。

## 影響

- `spec/v1/spec.md`: §5 の数値項に `-0` の域内正規化（`0` と同一 identity、MUST）を追加。§2 の互換ポリシーに第 2 の据え置き類型（全実装が既に一致した挙動の規範化）を追加。
- `docs/adr/0010-*.md`: specVersion 据え置き条件の精緻化を改訂注記で記録。
- `testdata/v1/id-vectors.json`: `negzero`（`-0` → canonical `0`）の受理ベクタを追加。
- `docs/design.md` の ADR 索引、`CONTEXT.md`（identity の値域 項）を同期。
- `go/id.go` / `nix/lib.nix`: **変更なし**。既存挙動が本 ADR の規範と一致するため。
- id 導出のアルゴリズム・安定性契約・key の最小性（ADR-0004）、identity 値域の締め出し方針（ADR-0024）は不変。

## 棄却した代替案

- **`-0` を域外拒否とする**: 整数の `-0` は `0` と同一で割れる余地がないため、拒否は実装を複雑化するだけで互換上の利得がない。両実装の現行の素直な正規化経路を捨てることになる。
- **`-0` を `0` と別 identity として保持する**: 整数表記に符号付きゼロの区別を持ち込むことになり、`builtins.fromJSON` / `encoding/json` のどちらも `-0` を `0` に畳むため、そもそも実装で区別を保持できない。規範化しても機械的一致が取れない。
- **spec に明文化せず実装挙動に委ねる**: 暗黙合意のまま id-vectors にベクタだけ足すと、将来 `fromJSON` / `encoding/json` の挙動が変わったとき何が正なのか判断できない。id-vectors の言語間互換の存在意義（域を規範で固定する）に反する。
