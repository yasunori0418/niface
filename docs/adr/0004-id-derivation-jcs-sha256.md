# ADR-0004: item id を sha256(JCS(identity)) で機械導出し安定性を保証する

- ステータス: 採用
- 日付: 2026-07-05
- 関連: `spec/v1/spec.md`（§5）, `testdata/v1/id-vectors.json`, `go/id.go`, `nix/lib.nix`

## 背景

item を世代・実行・plan/apply を跨いで同定する id が要る。「`種別:キー`」の文字列規約は、エスケープ・文字種・区切り文字衝突といった形式上の問題をツール実装者に押し付ける。

## 決定

- `identity = { "kind": <string>, "key": <JSON value> }` とし、 `id = lowercase-hex( sha256( JCS( identity ) ) )` の 64 文字 hex で導出する。 JCS は RFC 8785（JSON Canonicalization Scheme）。
- ツールが定めるのは identity の中身のみ。`key` は対象の**宣言上の同一性を表す最小の値集合**とし、ビルド毎・実行毎に変わる値（store パス・タイムスタンプ・世代番号）を含めない。
- 同一対象の id は世代・実行・plan/apply を跨いで**不変**とする（違反はツールの欠陥）。
- 可読性を補うため `kind` を必須フィールド化し、`label` を推奨する。

## 根拠

- エスケープ・文字種・一意性という形式上の問題を規格が吸収し、ツールに残る判断を **key の選定だけ**に帰着させる。
- 安定性契約も「key にビルド毎に変わる値を含めない」という**検証可能な規則**に単純化される。
- JCS で正準化してから hash するため、フィールド順や表現の揺れに依存しない。

## 影響

- `spec/v1/spec.md` §5。
- `testdata/v1/id-vectors.json`（identity → 期待 id の対応表）が言語間互換の要。 JCS の罠（非 ASCII キー・数値表現・ネスト）を突くベクタを含め、全言語実装がこれを通すことで導出の互換を証明する。
- `go/id.go` / `nix/lib.nix` の導出実装。

## 棄却した代替案

- **「`種別:キー`」の文字列規約**: エスケープ・区切り文字衝突・文字種の問題をツール実装者へ転嫁する。
- **人間可読な id 文字列**: 一意性と安定性を規格として保証しにくい。可読性は `kind` 必須 + `label` 推奨で別途担保する。
