# ADR の書式と改訂注記の運用

niface の設計判断は 1 件 = 1 ADR（Architecture Decision Record）として `docs/adr/` に記録する。決定の根拠と**棄却した代替案**を残し、後から「なぜこうなっているか」「なぜ別案にしなかったか」を追えるようにする。

規範仕様は `spec/v1/spec.md`、用語は `CONTEXT.md`、設計の概要と索引は `docs/design.md` にある。ADR はそれらの背後にある**判断の記録**を担う。

## 書式

各 ADR は連番ファイル `NNNN-<kebab-case-title>.md` とし、次の構成に従う。

```markdown
# ADR-NNNN: <決定を一文で>

- ステータス: 採用
- 日付: YYYY-MM-DD
- 関連: `spec/v1/spec.md`（§n）, ADR-XXXX 等
- 改訂対象: ADR-XXXX §n の〜を〜（既存 ADR に触れる場合のみ）

## 背景
<なぜこの判断が要るか。制約・前提>

## 決定
<何をどう決めたか。箇条書きで簡潔に>

## 根拠
<なぜこの決定が最良か>

## 影響
<spec / schema / testdata / 実装のどこに波及するか>

## 棄却した代替案
<検討したが採らなかった案と、採らなかった理由>
```

## 改訂注記の運用（supersede しない）

ADR は **supersede しない**。後続 ADR が既存 ADR の決定を反転・精緻化・拡張したときも、旧 ADR のステータスは「採用」のまま残し、旧 ADR 本文に blockquote 注記（back-reference）を追記して読者を後続 ADR へ誘導する。旧 ADR だけを読んで上書き済みの内容を最新仕様と誤認するのを防ぐ。

### いつ書くか

新しい ADR を書き、その中で既存 ADR の決定に触れる場合、**同じ変更の中で** 対象の旧 ADR 側にも注記を追記する。「後で追記する」は崩壊の原因になるため許容しない。新 ADR のヘッダに `改訂対象:` 行を書き、そこに挙げた ADR 全てに同じコミットで注記を書き戻す。

### 書式

対象 ADR のメタデータ行と `## 背景` の間に、時系列順で blockquote を積む。

```markdown
> **YYYY-MM-DD 改訂注記（ADR-XXXX）**: 本 ADR の何（§節番号があれば明記）が対象か。
> 何が不変で、何がどう変わったか（→ ADR-XXXX）。
```

- **日付**は改訂した側（新しい）ADR の日付。
- 「決定 X は不変。ただし Y は Z に変わった」のように、**生きている部分**と **上書きされた部分**を両方明示する。
- **ステータス行（`- ステータス: 採用`）は変更しない**。

### セルフチェック

新しい ADR を書き終えたら、ヘッダの `改訂対象:` に挙げた旧 ADR 全てに、その ADR 番号が現れているか確認する。

```sh
grep -rl "ADR-<新ADR番号>" docs/adr/*.md
```

挙げた旧 ADR がこの結果に出てこなければ注記漏れ。同じ変更の中で追記してから完了とする。
