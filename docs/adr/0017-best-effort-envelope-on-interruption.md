# ADR-0017: 中断時は best-effort でエンベロープを出力する

- ステータス: 採用
- 日付: 2026-07-07
- 関連: `spec/v1/spec.md`（§1）, ADR-0005, ADR-0008, ADR-0016

## 背景

SIGINT / SIGTERM 等による実行の中断を spec は何も規定しておらず、中断されたツールが stdout に何も出さない・不完全な JSON を残す、のどちらも許される状態だった。中断時に何も残らないと、それまでに適用された変更が消費側から不可視になり、ADR-0016 が確立した「適用済み差分の完全性」が中断のケースで穴になる。（tracking #8 / #12）

## 決定

- 中断されたツールは、可能な限り、中断時点までの `items` と適用済みの `changes` を含む valid なエンベロープを stdout に出力すべきである（SHOULD）。
- この場合の `status` は `error`、exit code は非 0 とする。中断専用の exit code 体系は作らない（POSIX 慣行 → ADR-0008）。
- 中断専用の共通エラーコード（`E_INTERRUPTED`）は追加しない。中断の報告にはツール別コード（`E_<TOOL>_INTERRUPTED` 等）または既存の共通コードを使う。

## 根拠

- SIGKILL・電源断・OOM kill ではプロセスに出力の機会が無く、MUST は物理的に守れない。守れない規範を MUST にすると適合判定が壊れる。SHOULD なら「トラップ可能な中断では出す」という実装可能な期待を規範にできる。
- best-effort でも中断時点までの items / 適用済み changes が構造化されて残れば、rollback・監査・再実行の判断が「何も残らない」場合より確実になる。中断を検知してから単一文書を出す動きは stdout 単一文書原則（§1）とも整合する。
- 共通コードの追加は「2 つ以上のツールで同じ意味の code が必要」を条件とする needs 駆動（→ ADR-0005）であり、ツールが 1 つも完成していない現時点では未達。必要が実証されたら互換追加できる。

## 影響

- `spec/v1/spec.md` §1 に中断時の SHOULD を追記。
- `schema/v1/` / `testdata/v1/` / `go/`: 変更なし（中断時の出力は通常の error エンベロープと同形で、新しい形は生まれない）。
- `docs/design.md`（索引）: 同期。

## 棄却した代替案

- **MUST 化**: SIGKILL 等プロセスが即死する中断では出力自体が不可能で、守れない MUST になる。
- **`E_INTERRUPTED` の共通コード追加**: 共通コードの追加条件（2 ツール以上・needs 駆動 → ADR-0005）未達。凍結した初期レジストリを投機的に緩める前例を作らない。
- **中断時は何も出力しないと規定する**: 不完全な JSON の混入は防げるが、適用済み変更が不可視になり、ADR-0016 の完全性が中断のケースで失われる。
