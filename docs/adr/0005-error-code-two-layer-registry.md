# ADR-0005: エラーコードを二層命名にし共通レジストリを凍結する

- ステータス: 採用
- 日付: 2026-07-05
- 関連: `spec/v1/spec.md`（§3, §6）

## 背景

エラーの機械識別子（`code`）が要る。エコシステムには複数ツールにまたがる共通概念と、特定ツールの概念でないと意味が取れないエラーが混在する。共通コードを最初から大量に定義すると、未使用コードの死蔵や意味の陳腐化を招く。

## 決定

- 二層命名とする。共通コード `E_<NAME>`、ツール別コード `E_<TOOL>_<NAME>`。警告も同様に `W_` prefix。
- 共通コードは初期 9 個で凍結する: `E_INPUT` / `E_SPEC_VERSION` / `E_PERMISSION` / `E_IO` / `E_LOCK` / `E_NOTFOUND` / `E_PRECONDITION` / `E_UNSUPPORTED` / `E_INTERNAL`。
- 共通コードの追加は「2 つ以上のツールで同じ意味の code が必要になった」ことを条件とする（needs 駆動）。追加は互換変更。

## 根拠

- 他のツールでも同じ意味で出しうるなら共通、ツールの概念を知らないと意味が取れないならツール別、という明確な判断基準を持てる。
- needs 駆動の追加条件により、共通レジストリの早すぎる肥大と「1 ツールしか使わない共通コード」の発生を防ぐ。

## 影響

- `spec/v1/spec.md` §3（Error / Warning）・§6（エラーコード + 共通コード初期レジストリ）。

## 棄却した代替案

- **数値コード**: 意味が不透明で、衝突管理と可読性の両方で劣る。
- **最初から大量の共通コードを定義**: 未使用コードの死蔵と意味の陳腐化を招く。needs 駆動で足りる。
- **単層命名（共通とツール別を区別しない）**: 消費側が「このコードは規格共通か個別ツール固有か」を判別できない。
