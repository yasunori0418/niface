# ADR-0022: 機微情報を info / detail に生で載せない SHOULD を記録する

- ステータス: 採用
- 日付: 2026-07-10
- 関連: `spec/v1/spec.md`（§3, §4）, ADR-0007, ADR-0006, ADR-0016, ADR-0003, ADR-0018, ADR-0019

## 背景

`info` / `detail` はツール固有情報の唯一の置き場所であり（→ ADR-0007）、規格はその中身を規定しない。しかし中身が無制約だと、ツールが実行結果を説明する過程で機微な値をそのまま載せてしまう経路が生まれる。例えば nshadow はユーザーのパスワード変更を扱うため `change.info` に crypt 形式のハッシュを old / new として載せたくなる。ncompose は世代の差分をログ・アーカイブとして永続化するため、`info` に載った値がエンベロープの寿命を超えて残る。どちらも消費側・保存先が信頼境界の外にあり得るため、機微値の生出力は情報漏洩の経路になる。

## 決定

- `spec/v1/spec.md` §3 に新節「機微情報の扱い」を設け、エンベロープ内の全 `info` / `detail` スロット（最上位 `info`、`result.info`、`item.info`、`change.info`、`error.detail`、`warning.detail`）に横断適用する SHOULD を記録する。
- 秘密・資格情報（パスワードハッシュ・秘密鍵・トークン等、非網羅）は生の値ではなくマスク・ダイジェスト（ハッシュ）参照・省略のいずれかで表すべきである（SHOULD）。具体的な表現手法は規定せず、ツールが文脈に応じて選ぶ。
- 既にハッシュ化された値（例: crypt 形式のパスワードハッシュ）であっても、それ自体が機微であれば同様に扱うべきである（SHOULD）。「ハッシュ済みだから安全」という誤解を先回りで否定する。
- `change.info` の機微値をマスク・省略しても `change.reversible`（§4）の値は変わらないことを明記する。可逆性はその遷移についてのツールの判断であり、公開された `info` 内の old / new 値ではなく特権的・帯域外なアクセスに依拠してよい。
- 何が機微に当たるかはツールの文脈依存とし、規格は網羅リストを持たない。

## 根拠

- `info` / `detail` の中身は元々ツール固有領域であり規格が構造を強制しない（→ ADR-0007, ADR-0018）。機微性の判断もツールの文脈に依存するため、値の中身に踏み込む制約は SHOULD が適切な強さで、MUST にすると機械検証不能な規範を課すことになる。
- reversible との切り離しを明記しないと、「機微値を隠す ⇒ 遷移を追跡できない ⇒ reversible: false にすべき」という誤読と、逆に「reversible: true を保つには機微値を生で出す必要がある」という誤読の両方が起こり得る。ADR-0003 の「可逆性は行為の属性」という決定は info の中身の可視性とは独立しており、本節はその独立性を明文化するだけで §4 本文は変更しない。
- nshadow / ncompose の実例が示すように、機微値の生出力は単発の表示問題ではなくログ・アーカイブへの永続化を通じて漏洩範囲が広がる。横断 SHOULD として 1 箇所に置くことで、個別スロットごとに同じ注意書きを重複させずに済む。

## 影響

- `spec/v1/spec.md` §3 末尾に新節を追加。§4〜§8 のトップレベル番号は変更しない。
- `CONTEXT.md` の `info` エントリの `_Avoid_` に本 ADR への参照を追加。
- `docs/design.md` の ADR 索引表に本 ADR の行を追加。
- schema / testdata / go / nix は無改訂（値の中身は元々 schema が規定しない領域のため）。

## 棄却した代替案

- **MUST 化する**: 機微性はツールの文脈依存であり、規格側に判定基準がない。schema の `additionalProperties: false` のような機械検証も効かないため、MUST にしても実効性のない規範になる。
- **正準マスク表現を規定する（例: `"***"` 固定・特定のハッシュアルゴリズム指定）**: `info` / `detail` の中身をツールが自由に設計できるという既存方針（→ ADR-0007）と矛盾する過剰制約になる。手法の選択はツールに委ねる。
- **対象を一般 PII まで拡張する（username・ホームディレクトリパス等）**: 「機微」の外延が曖昧になり SHOULD の適用範囲が判断不能になる。対象は開示が成りすましを可能にする、またはそれ自体が秘密である資格情報に限定する。
