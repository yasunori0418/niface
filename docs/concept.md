# niface: コンセプト

## これは何か

niface は、Nix ベースのシステム管理ツール群（nput / nboot / nwrap / nherd /
nshadow / ncompose）が **stdout / stdin を通じて会話するための共通 JSON 規格**である。
各ツールが単一の責務を持ち、パイプで組み合わさってディストリビューションを形成する
UNIX 哲学的なエコシステムにおいて、この規格は「配管の口径」を定義する。

## なぜ必要か

エコシステムの北極星は、NixOS とは異なる「Nix 版 Arch / Gentoo」——
最小のコアとユーザーによる組み立てを旨とするディストリビューションである。
NixOS がモジュールシステムという単一の大きな抽象で全体を統合するのに対し、
このエコシステムは**小さなツールの合成**で同じ問題を解く。

合成が成立する条件は、ツール間の契約が明確であることだ。契約が曖昧なら、
オーケストレータ（ncompose）はツール毎の場合分けに埋もれ、
「小さなツールの合成」は「密結合の分散」になってしまう。niface は
この契約を**ツールの実装から独立した規格**として管理し、
規格に適合しさえすれば、どの言語で書かれたどのツールでもパイプラインに
乗れることを保証する。

## 原則

1. **規格はツールに先行する**: ツールの実装都合が規格を歪めるのではなく、
   規格への適合がツールの受け入れ条件になる
2. **standalone 性の保証**: 規格に適合するツールは、単体で・どのディストロ上でも・
   JSON を手で与えて試せる。入力は stdin の JSON か明示引数のみとし、
   状態や設定の暗黙探索をしない
3. **意味は JSON が運び、成否は exit code が運ぶ**: exit code は POSIX 慣行のまま
   （0 / 非 0）とし、独自の意味体系を発明しない。何がどう失敗したかは
   エンベロープが構造化して伝える
4. **機械的に決まることを実装者に委ねない**: item id の導出のように
   決定的アルゴリズムで定義できるものは規格が定義し、
   ツール実装者に残る判断を最小化する
5. **規格型は閉じ、ツールの自由は隔離する**: ツール固有の情報は `info` 配下にのみ
   存在する。規格部分はツールを知らない消費者が型安全に扱える

## これがしないこと

- **ツールの実装を提供しない**: 参照実装・テストベクタは適合支援であり、本体ではない
- **オーケストレーションの方針を定めない**: パイプラインの順序・失敗時方針は
  ncompose とユーザーの領分。規格はその判断材料（status / changes / reversible）を
  運ぶだけ
- **ツール固有スキーマ（info の中身）を管理しない**: 各ツールが自リポジトリで
  管理する。規格がツールのリリースサイクルに引きずられることを避ける
