# エコシステム命名

計測日: 2026-07-05。衝突数は GitHub リポジトリ検索（`<name> in:name`、部分一致）の
件数で、正確な占有状況ではなく「埋もれやすさ」の近似シグナルとして扱う。

## 命名基準

nput 自身の命名判断（生の衝突数よりも「アクティブなプロジェクトかつ
Nix エコシステムという立ち位置でユニークか」）を基準として全体に適用する:

1. **役割の自明性**: 名前から責務が読めること（`n<役割>` パターンの一貫性）
2. **立ち位置のユニーク性**: 同名のアクティブ・著名プロジェクトと競合しないこと。
   休眠リポジトリとの部分一致は許容
3. **エコシステムの結束**: 傘名（未確定）と各ツールが 1 つの世界観
   （配管・束ねる）に収まると望ましいが、役割の自明性を犠牲にしない
4. **レジストリ名前空間**: 将来の配布先（crates.io / PyPI / pkg.go.dev）の
   空き状況も採用前に確認する

### nsvc 後継の検討（systemd 非依存が条件）

実測（2026-07-05）:

| 候補 | 衝突 | 評価 |
|------|------|------|
| **nherd** | 2 | **推奨**。herd = 群れの世話。GNU Shepherd の `herd` コマンドという先例により「サービス管理 = herd」の語彙が init 系文化に存在し、特定 init 非依存の含意が自然に出る。役割の自明性・短さ・空きの三拍子 |
| nsteward | 0 | 完全ユニークだが「何の steward か」が名前から読めない（基準 1 で劣後） |
| ndaemon | 2 | 役割は直球だが、ツール自体が常駐デーモンと誤解されるリスク（実体はワンショット CLI） |
| nrestart | 2 | reload / skip 判定も含む責務に対して狭い |
| nkeep | 25 | 埋もれる |
| ntend | 34 | 埋もれる |

### 規格名の追加検討（2026-07-05 追記・niface 確定の経緯）

| 候補 | 衝突 | 評価 |
|------|------|------|
| **niface** | **0** | **採用**。n + interface で規格の本質を直示。6 文字・読み（エヌアイフェイス / ニフェイス）の同音衝突なし。ただし傘 / ディストロ名への拡張性は弱いため規格名に限定 |
| nchord | 7 | 和音 = ツールの調和。ncompose と音楽語彙圏で揃う。先約は Chord DHT 実装（休眠気味）。「エヌコード」読みが なろう N コード・エンコードと同音になる問題 |
| nnchord | 0 | typo に見える・日本語 IME で nn = ん の入力摩擦・説明を要する |
| nif | 17,570 | 不採用。Erlang/Elixir NIF（現役の中核概念）・Gamebryo NIF 形式（MOD 界隈）と正面衝突 |
| nensemble | 0 | 独立した奏者の合奏 = 構図に最も近いが、長さの改善幅が小さい |
| ntutti | 2 | tutti = 全員で。知名度が低い |

命名の層の整理: **層が違う対象に別名を付けるのは健全**（Arch Linux / pacman / AUR が
別名であるのと同じ）。規格 = niface（確定）、ツール = n<役割>（確定）、
傘 / ディストロ = 北極星に近づいてから確定。

## 旧・実測結果と提案（記録として保持）

### 傘 / 規格

| 名前 | 衝突 | 判定 |
|------|------|------|
| **nmanifold** | **0** | **採用推奨**。多岐管 = 複数の管を 1 つの系に束ねる部品。エコシステム傘・ディストロ名・規格リポジトリを 1 語で張れる |
| nferrule | 0 | 次点。継手金具 = 接続規格の含意は最も正確だが、語の知名度が低く傘名に広げにくい |
| npipe 系 | 多数 | 不採用（named pipe 実装で飽和） |

### ツール群

| 現行案 | 衝突 | 提案 | 衝突 | 判断 |
|--------|------|------|------|------|
| nput | 115 | **nput のまま** | — | 確立済み。基準 2 を既に満たす |
| nboot | 28 | **nboot のまま** | — | 役割の自明性が最優先の層（ブート破壊は事故が重い）。netboot 系との部分一致は多いが完全一致の占有次第で許容。代替: nberth (6) |
| nwrap | 14 | **nwrap のまま**（代替: nsuid） | 1 | nwrap は capabilities も扱うため suid に限定しない現行名が正確。ほぼ空きの nsuid は将来 setuid 専用に分割する場合に温存 |
| nsvc | 19 | **nsysd** | 2 | nsvc は既存の service 系ツールと部分一致が多い。責務が systemd unit の差分適用に固定されているため、systemd を明示する nsysd の方が役割も立ち位置も明確。sd-switch アダプタに留まる場合は改名不要 |
| nuser | 100 | **nshadow** | 14 | nuser は一般語すぎて埋もれる（基準 2 を満たさない）。nshadow は駆動対象の shadow-utils / /etc/shadow を直接指し、役割の自明性と立ち位置の両方で勝る |
| ncompose | 6 | **ncompose のまま**（代替: nplait） | 0 | docker-compose の連想は「合成の指揮」の理解をむしろ助ける。完全ユニークにしたければ nplait（編む = 管を束ねる。傘の世界観とも合う）が空いている |

## 改名の要否まとめ

- **確定**: nput / nboot / nwrap / ncompose は現状維持、
  nsvc → **nherd**（systemd 非依存の命名が条件のため nsysd 案は棄却）、
  nuser → **nshadow**、規格 = **niface**
- **未確定**: 傘 / ディストロ名（候補: nmanifold / nchord。北極星に近づいてから確定）

## 採用前チェックリスト

各名前の採用確定前に以下を確認する:

1. GitHub 完全一致の占有: `gh api repos/<owner>/<name>` を著名 owner 数件と、
   `gh search repos --match name "<name>"` で完全一致のアクティブ度を目視
2. レジストリ空き: crates.io / PyPI / pkg.go.dev / npm の各名前空間
3. 商標・既存ディストロ名との衝突（傘 / ディストロ名候補のみ。DistroWatch の一覧を確認）
4. 検索性: `<name> nix` で検索したとき自プロジェクトが上位に来る余地があるか

## 備考

- ツール名は全て `n<役割>` の小文字 1 語で統一する（CLI 入力時の摩擦最小化）
- 傘名確定後は「<傘名> エコシステム」「(将来) <傘名> Linux」のように
  修飾語側で層を表現し、リポジトリ名の乱立を避ける。規格は niface として独立
